Milestone 6: Functional Prototype II

Changes since our last functional prototype

Since Milestone 4, we have made several significant changes to our prototype, which are each featured below:

  1. Change: Warn users that the instructor can monitor their behavior at all times, and that they can be seen/heard during the think phase
    Rationale: As part of the feedback we received from Milestone 5, John was especially concerned with user privacy regarding whether participants knew they were visible during the Think/Pair phases of the discussion. Though participants were informed of this verbally prior to participating in the user study, we wanted the same information to be communicated by default in the interface.
  2. Change: Improve messaging for Think/Pair/Share activities so students know what to expect before the discussion begins.
    Rationale: Currently, students didn’t quite know from the messaging what sort of activity they were about to participate in, and explaining the activity required the instructor reciting the same script prior to each study session. We wanted the activity to be more readily understandable from the interface itself.
  3. Change: Extend time for the pair phase (now set to 120 seconds by default, which is twice what it was before)
    Rationale: In our user studies, we found that participants were consistently unsatisfied with the length of time provided for the “Pair” phase of the discussion, in which each participant speaks with one other person within the discussion. Previously, the pair phase was set at 60 seconds, but participants had a difficult time introducing themselves and switching off in sharing their perspectives in a balanced way during the allotted time. To give students more time in discussing the prompt in pairs, we doubled the default amount of time dedicated to the “Pair” discussion activity.
  4. Change: Instructor's graph now supports an arbitrary number of students
    Rationale: Originally this was a pre-set number and our graph was statically generated. Now the Highcharts API call is inside a wrapper function that allows us to set style attributes and the number of users dynamically. This is half of the functionality needed to support arbitrary numbers of students in a group discussion (last week we could only support four).
  5. Change: Dynamic rather than pre-defined thresholds to trigger alerts
    Rationale: Thresholds for high or low participation are now dependent on the number of users in the Hangout, and adjusted thresholds are reflected in the instructor's graph and in the logic that triggers a notification.
  6. Change: Notifications only triggered once every 30 seconds and are under instructor control
    Rationale: User testing indicated that users disliked receiving constant notifications and desired spacing between these alerts. We have also routed all notifications to come from the instructor rather than the app itself, meaning that while not yet implemented, instructors will soon be able to approve/deny notificaitons sent to particular users and initiate new ones. Further user testing we will examine how best to address this interface.
